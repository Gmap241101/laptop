회원가입 정책·계정 복구·탈퇴 기능 배포 안내

기준 패키지
- member_directory_signup_policy_deployment_package.zip

배포 순서
1. deployment/src 내부 소스를 프로젝트의 src 폴더에 교체합니다.
2. deployment/firestore.rules를 Firebase 프로젝트의 firestore.rules 전체 교체본으로 배포합니다.
3. Firebase Authentication에서 이메일/비밀번호 로그인을 사용 설정합니다.
4. Firebase Authentication 비밀번호 정책을 최소 8자, 영문과 숫자 포함 기준으로 설정합니다.
   - 클라이언트 코드에서도 같은 기준을 검사하지만 비밀번호 원문은 Firestore Rules에서 검사할 수 없습니다.
5. 최초 배포 후 관리자 모드 > 부서·사용자 관리에서 명부를 저장하거나,
   관리자 모드 > 회원가입 정책에서 '전체 회원 명부 검사'를 한 번 실행합니다.
   - memberIdentityClaims와 accountRecoveryKeys를 기존 회원 기준으로 준비합니다.
   - 준비 전에는 기존 회원과 같은 부서·성명으로 중복 가입되는 것을 막기 위해 신규 회원가입이 비활성화됩니다.
6. 관리자 모드 > 회원가입 정책에서 아래 설정을 확인하고 저장합니다.
   - 등록된 부서·사용자만 가입 허용
   - 신규 회원 자동 승인
   - 자동 승인은 등록 명부 확인이 켜져 있을 때만 사용할 수 있습니다.
7. 필요하면 '전체 회원 명부 검사'를 수동 실행하여 기존 회원을 현재 명부 버전으로 일괄 확인합니다.

주요 동작
- 신규 가입 명부 확인은 최종 회원가입 버튼을 누를 때만 수행합니다.
- 가입 성공, 승인 대기, 이용 중지, 이용 종료, 비밀번호 재설정, 탈퇴 완료는 전용 상태 페이지로 표시합니다.
- 이메일 찾기는 부서·성명·연락처가 일치할 때 마스킹된 이메일만 표시합니다.
- 신규 회원 자동 승인이 켜져 있어도 탈퇴 후 재가입한 계정은 승인 대기로 생성됩니다.
- 기존 회원은 memberDirectoryVersion이 바뀐 경우에만 로그인 시 본인 명부 문서 1건을 재검증합니다.
- 전체 회원 검사는 자동 실행하지 않으며 관리자 버튼으로만 실행합니다.
- 회원 탈퇴 전 진행 중 신청·대여, 현재 연체, 유효한 대여 제한, 미처리 분실·파손 상태를 확인합니다.
- 탈퇴 후에도 memberIdentityClaims의 formerUids와 restrictionSnapshot을 보존하여 재가입 계정에 과거 UID와 제재 이력을 연결합니다.
- 관리자 승인 전 재가입 계정의 과거 UID에 진행 중 신청·대여가 남아 있으면 승인을 차단합니다.

Firestore Rules 관련 주의
- Rules는 명시적인 rentalRestrictions 문서의 차단·제재 필드를 확인하여 탈퇴를 제한합니다.
- 진행 중인 rentalRequests 전체를 Rules에서 임의 질의할 수는 없으므로, 신청·대여·연체 목록 확인은 앱의 탈퇴 트랜잭션 직전 검사와 함께 동작합니다.
- Spark 요금제의 순수 클라이언트 구조에서 서버 측 강제 탈퇴 워크플로는 제공하지 않습니다.
- 관리자가 다른 사용자의 Firebase Authentication 계정을 삭제하려면 Firebase Console을 사용해야 합니다.

검증 방식
- JavaScript/JSX 전체 구문 검사
- 로컬 import 경로 검사
- 컴포넌트 ctx 전달 검사
- 이메일 마스킹, 국내 전화번호, SHA-256 명부·복구키 유틸리티 검사
- Firestore Rules 괄호 및 컬렉션 중복 정의 검사
- ZIP 무결성 검사

실제 배포 전 권장 사항
- Firebase Emulator 또는 별도 테스트 프로젝트에서 다음 시나리오를 확인합니다.
  1. 수동 승인 가입
  2. 자동 승인 가입
  3. 재가입자의 자동 승인 제외
  4. 가입 대기·차단·종료 로그인 안내
  5. 이메일 찾기와 비밀번호 재설정
  6. 정상 탈퇴와 제한 상태 탈퇴 차단
  7. 탈퇴 후 재가입 이력 연결

시스템 관리 운영 콘솔 추가 배포 안내

추가 배포 순서
1. 최신 src 전체를 교체하고 최신 firestore.rules를 함께 배포합니다.
2. 최고 관리자 계정으로 로그인합니다.
3. 시스템 > 시스템 관리 > 사이트 기본 설정에서 사이트명·로고·색상·문의정보를 확인하고 저장합니다.
4. 서비스 운영 탭에서 정상 운영/읽기 전용/점검 중 상태와 기능별 접수 스위치를 확인합니다.
5. 관리자 보안 탭에서 무활동 로그아웃 시간과 1회 로그인 최대 유지시간을 저장합니다.
   - 보안정책 버전이 변경되면 기존 관리자 세션은 다시 로그인해야 합니다.
6. 기존 관리자 문서에 adminRole이 없으면 호환을 위해 최고 관리자(owner)로 해석됩니다.
   운영 안정화 후 관리자 ID 관리에서 일반 관리자(admin)와 최고 관리자(owner)를 구분하십시오.
7. 데이터 초기화 전 반드시 시스템 관리에서 개인정보 포함 백업을 내려받고 안전한 위치에 보관합니다.
8. 일반회원 Firestore 초기화 후 내려받은 UID 파일은 tools/delete-non-admin-auth-users.mjs 또는 Firebase Console을 이용해 Authentication 계정까지 정리합니다.

시스템 관리 주요 기능
- 사이트명, 기관명, 헤더 부제, 로고 URL, 파비콘, 브랜드 색상, 기본 비주얼, 문의정보
- 정상 운영/읽기 전용/점검 중 운영모드
- 회원가입, 신규 대여, 변경·취소, 연장, 반납 요청별 접수 스위치
- 시스템 상단 운영 안내 배너
- 관리자 무활동 세션 15~480분, 절대 최대 세션 1~24시간
- 최고 관리자/일반 관리자 역할과 최고 관리자 수동 계정 잠금
- 데이터 건전성 검사와 JSON 백업
- 자산, 일반회원, 신청·대여, 조직명부, 콘텐츠, 사이트설정 영역별 초기화
- 테스트 운영 데이터 전체 초기화와 공장 초기화 범위 선택
- 초기화 작업 상태 저장 및 중단 작업 재실행
- 시스템 설정 변경 감사 로그

데이터 초기화 주의
- 관리자 계정 문서(adminAccounts)와 관리자 Firebase Authentication 계정은 초기화 대상에서 제외됩니다.
- 자산 또는 회원정보만 초기화하려는데 대여내역이 남아 있으면 참조 무결성을 위해 실행이 차단됩니다.
- 초기화 시작 시 서비스는 점검 모드로 전환되며, 작업 완료 후 관리자가 정상 운영으로 직접 전환해야 합니다.
- Firestore 컬렉션은 문서 단위 배치 삭제로 처리됩니다. 삭제 문서 수만큼 읽기와 삭제 쓰기가 발생합니다.
- Firebase Authentication 일반회원은 브라우저 관리자 화면에서 타인 계정을 삭제할 수 없으므로 UID 파일과 로컬 Admin SDK 도구를 사용합니다.
- 서비스 계정 비공개 키는 소스, GitHub, Hosting, Firestore에 절대 저장하지 마십시오.

보안 관련 제한
- Firebase Authentication의 로그인 공격 보호를 사용합니다.
- 클라이언트만으로 신뢰할 수 있는 '비밀번호 5회 실패 후 계정 잠금'을 구현하지 않습니다.
- 시스템이 제공하는 수동 관리자 잠금은 최고 관리자만 실행할 수 있습니다.
